# Changelog — Junio 2026

## 2026-06-10

**[Feature]:** Cambio manual de estado de cotizaciones (reabrir para retrabajar).

Un dropdown de estado en `QuotationDetail` permite mover la cotización entre los 4
estados no terminales en cualquier momento (p. ej. `sent_for_approval → draft`), para
retrabajar sin recrear (clave en cotizaciones grandes).

- **`PATCH /api/quotations/[id]/status`** (nuevo) — valida que el target ∈
  {draft, sent_for_approval, approved, rejected} (`converted_to_order` no es destino
  manual → 400). Guarda: si la cotización está `converted_to_order`, consulta `orders`
  por `quotation_id`; si existe **cualquier** orden vinculada → 400 ("Elimina la orden
  vinculada…"). Exigir eliminación (no solo cancelación) garantiza ≤1 orden por
  cotización (evita órdenes huérfanas). **No toca `quotation_items`** → `is_approved`
  se preserva. Patrón calcado de `orders/[id]/status`.
- **`src/lib/quotation-status.ts`** (nuevo) — `QUOTATION_STATUS_LABELS` +
  `MANUAL_QUOTATION_STATUSES`. Reutilizado por `QuotationStatusBadge` (sin cambiar
  apariencia) y el dropdown.
- **`useChangeQuotationStatus`** (`useQuotations.ts`) — mutation vía `fetchJson`
  (hereda manejo 401/red de `ApiError`).
- **`QuotationDetail.tsx`** — `<Select>` de estado en el header + `AlertDialog` de
  confirmación. El dropdown se deshabilita cuando: hay cambios sin guardar (`isDirty`,
  hint "Guarda los cambios…") o la cotización está `converted_to_order` con orden
  vinculada existente (hint "Elimina la orden vinculada para reabrir"). Backend valida
  igual (defensa en profundidad).
- **Tests** (+5 API + 1 mock de componente) — target inválido, 404, revertir
  sent_for_approval→draft sin tocar items, guarda con orden activa (400), reabrir sin
  orden activa (200).

**Decisiones:** `converted_to_order` se revierte solo tras cancelar/eliminar la orden
(que ya restaura inventario vía `restoreOrderInventory`) → sin doble manejo de stock.
Las decisiones del cliente (`is_approved`) se preservan al revertir. El `approval_token`
no se regenera.

**Total:** 277 tests, 0 fallos.

**Motivo:** No había forma de regresar una cotización a un estado anterior; recrear una
cotización grande (cientos/miles de ítems) era inviable.

## 2026-06-09

**[Feature]:** Changelog "Novedades" visible en la app.

Página `/dashboard/changelog` que muestra una línea de tiempo de mejoras y correcciones
en lenguaje de usuario (no técnico). Accesible desde un botón "Novedades" en el Sidebar.

- **`CHANGELOG.md`** (raíz, nuevo) — fuente de verdad en español simple, formato Keep a
  Changelog (`## YYYY-MM-DD` → `### Nuevo|Mejorado|Corregido` → `- entrada`). Es la
  versión para usuarios; la bóveda `06-Changelog/` sigue siendo el detalle técnico del dev.
- **`src/lib/changelog.ts`** (nuevo) — `parseChangelog(raw)`: parser puro line-based →
  `ChangelogRelease[]` con fecha, versión opcional y entradas tipadas por categoría. Une
  líneas de continuación (wrap), ignora intro/`---`/título.
- **`src/app/dashboard/changelog/page.tsx`** (nuevo) — Server Component; lee el `.md` con
  `fs` (`process.cwd()`), renderiza timeline con el design system existente (cards +
  badges verde/azul/ámbar, reusa `docs-page-bg`/`login-card-border` de la docs page).
- **`src/components/layout/Sidebar.tsx`** — `NavLink` "Novedades" (icono `Sparkles`) junto
  a "Documentacion".
- **`next.config.ts`** — `outputFileTracingIncludes` para `/dashboard/changelog` → incluye
  `CHANGELOG.md` en el bundle de producción/Vercel (si no, `fs` fallaría en runtime).
- **`tests/lib/changelog.test.ts`** (+9) — parser: orden de releases, versión opcional,
  mapeo de categorías, wrap multi-línea, agrupación, intro ignorada, vacío → `[]`, `##`
  sin fecha no abre release, entrada sin categoría se ignora.
- **`CLAUDE.md`** — nueva fila en la tabla de auto-mejora: cambios visibles para el usuario
  se registran también en `CHANGELOG.md` raíz.

**Total:** 271 tests, 0 fallos.

**Motivo:** El desarrollador y los usuarios de DYMMSA no tenían un lugar dentro de la app
para ver qué se fue resolviendo/agregando. La bóveda Obsidian es solo para el dev y es
demasiado técnica.

## 2026-06-09 (otros cambios del día)

**[Fix]:** Límite de PostgREST truncaba cotizaciones/órdenes grandes.

- Las queries con relación embebida (`quotation_items(*)`, `order_items`) usaban el límite
  por defecto de PostgREST (1000 filas), truncando silenciosamente cotizaciones grandes.
  Al re-guardar (DELETE+INSERT) se perdían los ítems truncados.
- Detectado: cotización "DAÑADAS Y PERDIDAS PARTE 5" con exactamente 1000 ítems (debía
  tener ~1084).
- Fix: `.limit(3000, { foreignTable: 'quotation_items' })` en `useQuotation` y
  `create-order`; `.limit(3000)` en `useOrders`.

**[Fix]:** Aplicado feedback del PR reviewer (#9) en `QuotationDetail`.

- `handleSendForApproval` ahora usa `handleApiError` (redirect en `AUTH_EXPIRED` + highlight
  de fila ofensora) y pre-flight `getBlockingIssues` antes del auto-save.
- `SortableContext` IDs memoizados con `useMemo` (consistencia con `QuotationEditor`).
- `CLAUDE.md`: documentado que `canEdit` incluye `sent_for_approval`.

**[Feature]:** Display name en el saludo del dashboard.

- `dashboard/page.tsx` usa `user.user_metadata.full_name` con fallback al prefijo del
  correo. Nombres asignados a los 3 usuarios actuales vía SQL (`raw_user_meta_data`).

**Motivo:** Issues reportados por el usuario durante la sesión.
