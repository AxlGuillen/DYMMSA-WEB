# Changelog — Mayo 2026

## 2026-05-26

**[Test]:** Migración del testing de `bun:test` a **Vitest** (runner único).

Se consolidó en un solo runner que cubre backend y componentes, eliminando todo uso de `bun:test`. Bun se mantiene como package manager/runtime.

- **`vitest.config.ts`** (raíz) con `test.projects`: entorno `node` (`tests/lib` + `tests/api`) y `jsdom` (`tests/components`). Alias `@/` nativo (`resolve.tsconfigPaths`).
- Migrados los 178 tests existentes: imports `bun:test`→`vitest`, `mock`→`vi`, y el patrón de inyección pasó a `vi.mock('@/lib/supabase/server', () => ({ createClient: vi.fn() }))` + `vi.mocked(createClient).mockImplementation(...)` en `beforeEach` (permite imports estáticos).
- **Harness de componentes** nuevo: `tests/components/setup.ts` (jest-dom + cleanup + polyfills `matchMedia`/`ResizeObserver`) + `smoke.test.tsx` (render de `QuotationStatusBadge`) → valida jsdom + React 19.
- devDeps: `vitest`, `@vitest/coverage-v8`, `@vitejs/plugin-react`, `jsdom`, `@testing-library/{react,jest-dom,user-event}`.
- Scripts: `vitest run` / `vitest` / `vitest run --coverage`.
- `tests/tsconfig.json` para IntelliSense sin afectar el build (Next excluye `tests`).

**Total:** 180 tests, 0 fallos (~1.3 s).

**Gotcha:** usar `bun run test`, **no** `bun test` (el runner integrado de Bun no se puede deshabilitar y falla con imports de `vitest`).

**Motivo:** al decidir agregar tests de componentes (React 19 + jsdom), un solo runner es más simple que mantener `bun:test` + Vitest en paralelo. La suite era joven, así que migrar fue barato.

## 2026-05-24

**[Fix]:** No se podía convertir a orden una cotización con separadores (encabezados).

- Síntoma: al agregar separadores a una cotización y luego presionar "Generar Orden", la conversión fallaba silenciosamente (rollback) y nunca se creaba la orden.
- Causa: la migración `add_item_type_and_section_label` (2026-03-30) introdujo los separadores pero **no relajó** la constraint `order_items_quantity_approved_check` (`CHECK (quantity_approved > 0)`). `create-order` inserta cada separador con `quantity_approved = 0`, lo que violaba la constraint y abortaba el INSERT batch completo de `order_items`. El feature estuvo roto desde su origen (0 separadores en producción).
- Fix: migración `allow_separators_in_order_items_quantity` — la constraint ahora es `CHECK (quantity_approved > 0 OR item_type = 'separator')`. Los productos siguen exigiendo `> 0`; los separadores con `0` ya pasan. El invariante `quantity_in_stock + quantity_to_order = quantity_approved` se mantiene (0 = 0 + 0).

**Migración:** `allow_separators_in_order_items_quantity`
**Tabla:** `order_items`

## 2026-05-16

**[Fix]:** UI desincronizada al editar cotización aprobada.

Tras guardar una cotización en estado `approved` (reorden + separadores), los badges de aprobación caían a "Pendiente" hasta recargar la página.

- Causa: el route `PATCH /api/quotations/[id]/update` hace DELETE+INSERT regenerando IDs. El `useEffect` en `QuotationDetail.tsx` solo dependía de `[quotation.id]`, así que tras refetch los nuevos IDs no se propagaban a `localItems`.
- Fix: agregar `itemsSignature` (concatenación de IDs) como dep del useEffect — re-sincroniza tras save sin perder ediciones en refetches pasivos.

**Archivo:** `src/components/quotations/QuotationDetail.tsx` L417-422
