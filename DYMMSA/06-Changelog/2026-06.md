## 2026-06-19 (II)

**[Refactor/Security]:** Lecturas de cotizaciones migradas al server side (flujo crítico).

Cierra el patrón "no Supabase directo desde el cliente" en el flujo de cotizaciones. Las
escrituras ya iban por API routes; faltaban las **tres lecturas**, que consultaban Supabase
directo desde el navegador. (Quedan pendientes `useOrders`, `useDashboard` y `useProducts`,
que aún leen directo; `useAuth`/`login` seguirán usando el browser client por diseño.)

- **`GET /api/quotations/[id]`** (nuevo, agregado al archivo que ya tenía `DELETE`) —
  cotización + `quotation_items(*)` ordenados por `sort_order`, `limit(5000)` contra el
  truncamiento >1000, 404 si no existe. Es el read que alimenta el editor (`QuotationDetail`)
  → el más crítico. (La doc afirmaba que esta ruta existía; en realidad solo había `DELETE`.)
- **`GET /api/quotations`** (nuevo) — lista paginada con búsqueda `.or` (saneada), filtro
  `status` con whitelist, orden `created_at desc`, y **mapeo de `items_count`** en el server.
- **`GET /api/quotations/stats`** (nuevo) — reduce por status.
- **`useQuotations.ts`** — `useQuotations`/`useQuotationStats`/`useQuotation` reescritos con
  `fetchJson` (mismas firmas y mismas query keys → páginas/componentes y la invalidación de
  las mutations intactos). **Eliminado el import de `@/lib/supabase/client`** → el hook ya no
  toca Supabase directo.
- **Tests** (+10): `tests/api/quotations-read.test.ts` — single (embed, orden `sort_order`,
  `limit(5000)`, 404), list (shape, `.or`, whitelist de status, `items_count`), stats.

**Total:** 328 tests, 0 fallos.

**Motivo:** Consistencia y seguridad: las lecturas del flujo más importante dependían solo de
RLS y eran el último resto del patrón de acceso directo desde el cliente.

## 2026-06-10 (II)

**[Feature/Security]:** Regenerar `approval_token` al reabrir + preservar aprobaciones.

Cierra el hueco del link público de aprobación al reabrir cotizaciones.

- **`PATCH /api/quotations/[id]/status`** ahora regenera `approval_token`
  (`crypto.randomUUID()`) en cada cambio de estado. El link compartido previamente deja
  de matchear → la página `/approve/[token]` devuelve 404 (link muerto). Evita que un
  cliente apruebe con un link viejo tras una reapertura.
- **`PATCH /api/quotations/[id]/update`** ahora preserva `is_approved` en **cualquier**
  estado editable (antes solo en `approved`). Al reabrir y agregar ítems nuevos, las
  decisiones previas del cliente se conservan; los nuevos quedan `null` (pendientes).
  La página `/approve/[token]` ya pre-selecciona los `is_approved === true`, así que el
  cliente solo decide los nuevos.
- **`/approve/[token]/page.tsx`** — añadido `.limit(5000, { foreignTable })` al embed de
  `quotation_items` (mismo bug de truncamiento >1000 que el resto, en la vista del cliente).
- **Tests** (+2): status regenera token; update preserva `is_approved` en `draft`.

**Decisión:** Opción A (regenerar token) sobre congelar el link con una columna nueva —
cero cambios de schema. La preservación de aprobaciones evita que el cliente re-apruebe
todo al sumar ítems.

**Total:** 279 tests, 0 fallos.

**Motivo:** Reabrir una cotización a `sent_for_approval` reactivaba el mismo link; el
cliente podía re-decidir y pisar el trabajo. Además, editar una reabierta borraba las
aprobaciones (update solo preservaba en `approved`).

## 2026-06-10

**[Feature]:** Cambio manual de estado de cotizaciones (reabrir para retrabajar).

Un dropdown de estado en `QuotationDetail` permite mover la cotización entre los 4
estados no terminales en cualquier momento (p. ej. `sent_for_approval → draft`), para
retrabajar sin recrear (clave en cotizaciones grandes).

- **`PATCH /api/quotations/[id]/status`** (nuevo) — valida que el target ∈
  {draft, sent_for_approval, approved, rejected} (`converted_to_order` no es destino
  manual → 400). Guarda: si la cotización está `converted_to_order`, consulta `orders`
  por `quotation_id`; si existe **cualquier** orden vinculada → 400 ("Elimina la orden
  vinculada…"). Exigir eliminación (no solo cancelación) garantiza ≤1 orden por
  cotización (evita órdenes huérfanas). **No toca `quotation_items`** → `is_approved`
  se preserva. Patrón calcado de `orders/[id]/status`.
- **`src/lib/quotation-status.ts`** (nuevo) — `QUOTATION_STATUS_LABELS` +
  `MANUAL_QUOTATION_STATUSES`. Reutilizado por `QuotationStatusBadge` (sin cambiar
  apariencia) y el dropdown.
- **`useChangeQuotationStatus`** (`useQuotations.ts`) — mutation vía `fetchJson`
  (hereda manejo 401/red de `ApiError`).
- **`QuotationDetail.tsx`** — `<Select>` de estado en el header + `AlertDialog` de
  confirmación. El dropdown se deshabilita cuando: hay cambios sin guardar (`isDirty`,
  hint "Guarda los cambios…") o la cotización está `converted_to_order` con orden
  vinculada existente (hint "Elimina la orden vinculada para reabrir"). Backend valida
  igual (defensa en profundidad).
- **Tests** (+5 API + 1 mock de componente) — target inválido, 404, revertir
  sent_for_approval→draft sin tocar items, guarda con orden activa (400), reabrir sin
  orden activa (200).

**Decisiones:** `converted_to_order` se revierte solo tras cancelar/eliminar la orden
(que ya restaura inventario vía `restoreOrderInventory`) → sin doble manejo de stock.
Las decisiones del cliente (`is_approved`) se preservan al revertir. El `approval_token`
no se regenera.

**Total:** 277 tests, 0 fallos.

**Motivo:** No había forma de regresar una cotización a un estado anterior; recrear una
cotización grande (cientos/miles de ítems) era inviable.
